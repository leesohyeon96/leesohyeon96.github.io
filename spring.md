# Spring이란? 

**- JAVA의 웹 *프레임워크(Framework)로 JAVA언어를 기반으로 사용**
**- 개발을 더 쉽게 해주는 프로그램 틀**
<br/>

# * 프레임 워크(Framework)란?

**- 소프트웨어의 구체적인 부분에 해당하는 설계와 구현을 재사용이 가능하게끔 일련의 `'협업화된 형태로'` 클래스들을 제공하는 것 -by Ralph Johnson(랄프 존슨)-**

**- 개발하기 위해 지켜야 하는 '틀'이라고 보면 됨**

**- BootStrap(부트스트랩) 은 '프레임 워크의 한 종류'** 

  ​    

**- 자주 쓰일 만한 기능들을 한데 모아 놓은 클래스들의 모음!**

**- 설계의 기반이 되는 부분을 기술한 확장 가능한 '기반 코드' + 사용자 입맛대로 수정,확장 하는데 필요한 '라이브러리' => 통합되어 제공되는 형태**

**- 장점 : 비교적 빠른 시간에 완성 및 유지보수 가능**

**- 단점 : 학습 난이도가 높으며 개발자의 입김대로 코드를 작성할 수 없음** 

  ​     

# * 프레임 워크 vs 라이브러리 

**- 라이브러리 : 자주 사용되는 로직을 `'재사용하기 편리하도록'` 잘 정리한 `'일련의 코드들의 집합'`  => 클래스, 템플릿, 설정 데이터 등**

**- 라이브러리 장점** 

  **- 가독성/유지보수 향상 : 코드의 중복을 줄임**
  **- 생산성 향상 : 프로그래밍 시간 줄임**

  **ex) 프레임워크 : 자동차를 구성하고 있는 뼈대**

  **ex) 라이브러리 :  자동차의 기능을 하는 부품[ex. 바퀴, 헤드라이트 등]**

  

  **-> 한번 정해진 자동차의 뼈대(프레임워크)는 바꾸질 못함 but 부품(라이브러리) 는 비교적 쉽게 다른 종류로 바꿀 수 있음!**

**- 즉, 원하는 기능을 구현하기 위해 '기본적인 뼈대(프레임워크)'와 '부품(라이브러리)'를 가져다 쓰는 것!!**

**- 주된 차이 : `'제어의 역전(Inversion Of Control)'` 프레임워크에는 작업을 호출할 수 있는 '제어권'이 있다는 것! (사람이 할일을 프레임워크가 주도권을 가져가 대신 수행함!)**  

  <br/>

# * Spring의 핵심 3대 요소

**1) 제어의 역전(IOC, Inversion Of Control)**  
**2) 관점 지향 프로그래밍(AOP, Aspect Oriented Programming)**  
**3) 서비스 추상화(PSA, Portable Service Abstraction)** 
<br/>

## (1) 제어의 역전

**- 객체의 생성 ~ 소멸까지의 '객체 생명주기 관리를 개발자가 하는 것이 아닌 Spring 컨테이너가 대신 해주는 것'!**

**- Controller가 의존성을 사용하지만 직접 관리 X  =>  즉, 의존성 생성/관리를 하지 않음!**

  **- DL(Depnedency LockUp) : 저자옷에 저장되어 있는 Bean에 접근하기 위해서 `컨테이너가 제공하는 API를 이용하여 Bean을 LockUp하는 것!`** 

  **- DI(Dependency Injection) : `'의존성을 주입'`하는 역할을 하며, 프로그래밍에서 `'구성요소 간의 의존관계'`가 소스코드 내부가 아닌 `'외부의 설정파일을 통해 정의'`되는 방식!**   

  **- 쉽게 말하면 DI는 `필요한 부분을 꼇다 뺏다` 하면서 적절한 상황에 필요한 기능을 사용할 수 있게 해주는 것!**

    1. 수정자 주입(Setter Injection) : 의존성을 입력받는 Setter메소드를 만든 뒤 이를 통해 의존성 주입받음
       -> 주입받는 객체가 변경될 가능성이 있는 경우

    2. 생성자 주입(Constructor Injection) : 필요한 의존성을 포함하는 클래스의 생성자를 만들고 이를 통해 의존성 주입받음 -> 생성자 호출 시점에 1회 호출 되는 것이 보장됨!
 
    3. 필드 주입(Field Injection) : 변수 선언부에 @Autowired 를 사용하여 의존성 주입받음 

       - 간단하지만 여러가지 문제점이 있어 '생성자 주입'을 통한 DI를 권장함!

         *여러가지 문제점

             1) 단일 책임(SRP)의 원칙 위반
             2) 테스트 용이하지 않음
             3) final 선언 불가로 인한 불변성 보장X
             4) 순환 의존성에 대한 문제점 
             5) 외부에서 접근이 불가능함

       #### * SOLID 원칙(객체 지향의 5대 원칙)

       - 목적 : 프로그래머가 시간이 지나도 유지 보수와 확장이 쉬운 소프트웨어를 만드는 데 도움을 주기 위함!

         

       - 종류

       1. SRP(단일 책임 원칙)
          1. "클래스는 단 한 개의 책임을 져야 한다"
       2. OCP(개발-폐쇄 원칙)
       3. LSP(리스코프 치환 원칙)
       4. DIP(의존 역전 원칙)
       5. ISP(인터페이스 분리 원칙)

    

    

  - ### 생성자주입을 권장하는 이유?

    **- 생성자의 호출 시점에 1회 호출되는 게 보장**

    **- Spring 프레임워크에서 생성자 주입을 적극 지원하기 때문에 생성자가 1개만 있을 시 @Autowired 생략 가능**

    **- 객체의 불변성 확보 -> 생성자 주입 이외의 것들은 수정의 가능성을 열어두어 유지보수성을 떨어트림**

      **- *객체의 불변성 확보시 장점?**

        1. 실행 중인 쓰레드 간 서로의 간섭에 의해 생길 수 있는 '동기화 문제'에 대한 걱정을 안 해도 된다!( = Thread safe ! )		

           *한 쓰레드가 작업 중 다른 쓰레드에 의해 간섭을 받지 못하도록 막는 것을 '쓰레드 동기화'라고 한다. 

           *동기화 처리하기 위해 모든 객체에 '락(rock)'을 포함 시켰으며 이는 모든 객체가 힙 영역에 생성될 때 자동으로 만들어짐! 

        2. side-effect(부작용) 에 대한 걱정이 없기 때문에 설계, 구현 및 사용하는 데에 편리함

    **- 테스트 코드의 작성 -> 생성자 주입 이외의 코드들은 순수한 자바 코드로 단위 테스트를 작성하는 것이 어려움 + 생성자 주입은 컴파일 시점에 객체를 주입받아 테스트 코드를 작성할 수 있고 오류를 발견할 수 있음**

    **- final 키워드 작성 및 Lombox과의 결합 -> 컴파일 시점에 누락된 의존성을 확인할 수 있음 + 생성자 주입 이외의 것들은 객체생성(생성자 호출) 이후에 호출되므로 final 키워드를 사용할 수 없음 + final 키워드를 붙임으로써 Lombok과 결합되어 코드 간결화 가능**

    **- 순환 참조 에러 방지 -> 객체의 생성 시점(어플리케이션 구동 시점)에 순환 참조 에러를 예방할 수 있음**

**- Spring Ioc Container 란?**

  **- **`객체(Bean)의 생성을 책임`지고 `'의존성을 관리하는 컨테이너'`**
  **- 이로 인해, 개발자들은 비즈니스 로직에 집중할 수 있음**
  **- 스프링 컨테이너가 관리하는 객체는 '빈(Bean)'이라고 하고**
  **- 이 빈들을 관리한다는 의미로 컨테이너를 '빈 팩토리(BeanFactory)'라고 부름**
  **- 보통은 바로 BeanFactory 사용하지 않고, 이를 확장한 ApplicationContext(스프링의 각종 부가 기능을 추가로 제공함)를 사용함!**

## (2) 관점 지향 프로그래밍

- **AOP 사용하는 이유? proxy(기존 코드에 변화 없이 접근 제어 or 부가 기능 추가를 위해 사용)의 중복 코드가 생기고, proxy클래스를 만드는 데 들어가는 시간과 비용이 들기 때문에! **

**- 로깅, 트랜잭션, 보안 등 여러 모듈에서 `공통적으로 사용하는 기능을 분리`하여 관리할 수 있다.**

**- 중복되는 코드와 기능이 있는 클래스들은 실용성과 가독성 및 개발속도에 좋지 않음**

**- 즉, AOP는 `여러 객체에 공통적으로 적용할 수 있는 기능을 구분함으로써 재사용성을 높여주는 프로그래밍 기법` + 흩어진 `Asepct`를 모아서 모듈화 하는 프로그래밍 기법**

**- Spring AOP는 `개발자가 마주한 공통적인 문제를 해결`하고자 Spring Ioc를 통한 간단한 AOP구현이 그 목적!  완전한 AOP를 의도한 것이 아니며 => Spring 컨테이너에 의해 관리되는 Beans에만 적용이 가능함!**

  ```
  - 다수의 모듈에서 '반복적으로 나타나는 부분'을 횡단관심이라고 함! 
  - 핵심 기능이 아닌 중간중간 삽입되어야 할 기능들!  
  ```

  ```java
  ex)
  try {    
  
  insert / update / delete / select 실행 -> //핵심관심사항

  			//logging도 횡단관심사항이라고 볼 수 있다?
  } catch ... {
  
  } finally {
  
  }
  ```

  ​    *Target 

  ``` 
  - Aspect가 가지고 있는 Advice가 적용되는 대상(클래스, 메소드 등등)
  ```

  

  ​	*Aspect란? 

  	- 여러군데(객체)에 공통적으로 적용되는 관심 사항(중복되는 코드)
  	- 언제(Advice) + 어디서들(PointCut) => 어떤 pointcut 메소드에 대해서 어떤 advice 메소드를 실행할지 결정함
  	- 이 Aspect설정에 의해 AOP 동작 방식이 결정됨!
  	- advice가 target object에 삽입되서 동작하는 작업을 *weaving이라고 함

  ​	*AspectJ란?

  ``` 
  - AOP를 실현하는 '라이브러리'
  - Spring AOP에서 이 AspectJ를 이용한 AOP처리를 함 
  - AspectJ는 Spring AOP와 목적이 다름!
  - '완전한 AOP를 제공하는 것이 목적'인 근원적인 AOP기술
  ```

  

  ​	*pointcut란?

  ```java
  <aop:pointcut id="getPoincut" expression="execution(* com.springboot.biz...*Impl.get*(..))"/>
  - joinpoint의 정규 표현식
  - jointpoint가 pointcut에 일치할 때마다 해당 Pointcut에 관련된 Advice가 실행됨
  ```

  - 약간 필터같은 느낌

  

  ​		*Advice란?

  ```
  - 횡단 관심에 해당하는 공통 기능의 코드를 의미
  - '독립된 클래스의 메소드'로 작성됨
  - Advice의 동작시점[5가지]
    - before : 메소드 실행 전 동작 
    - after : 메소드 실행 후 무조건 동작
    - after-returning : 메소드가 성공적으로 리턴되면 동작
    - after-throwing : 메소드 실행 중 예외가 발생하는 시점에 동작
    - around : 메소드 호출을 가로채 실행 전&후&예외 발생 시점에 공통 기능 동작(이때, 반드시 ProceedingJoinPoint를 반드시 첫 번째 파라미터로 지정해야 함)
  ```

  

  ​	*weaving(위빙)이란? -> 실행될때를 말하는듯?

  		- pointcut으로 지정한 핵심 관심 메소드가 호출될 때, 		Advice에 해당하는 횡단 관심 메소드가 삽입되는 과정을 		의미함[Advice를 비즈니스 로직 코드에 삽입하는 것을 의미]
  		
  		- 이 weaving을 통해 비즈니스 메소드를 수정하지 않고도 횡	단 관심에 해당하는 기능을 추가하거나 변경할 수 있음 
  		
  		- 종류[3가지]
  	    	- 컴파일 시
  	    	- 클래스 로딩 시 
  	    	- 런타임 시(spring에서는 런타임 시만 지원)
  	    	  : Aspect가 대상 객체의 Proxy를 사용하는 
  	    	   애플리케이션의 실행 시에 Weaving 됨!
  		
  		- AOP에서는 핵심 로직에 직접 공통관심기능을 호출하지 않고 weaving을 통해서 공통관심기능 코드를 삽입함

  ​	*Proxy란? [Proxy(대리자/대변인) Pattern]

  ```
  - 대상 객체에 'Advice가 적용된 후 생성된 객체'를 의미
  - 비즈니스 로직을 구현한 객체에 직접 접근하지 않고 Proxy를 통해 접근함! 
  - 중요한 것은, '흐름제어'만 할 뿐 결과값을 변경시키면 안됨!
  - 장점 
  	- 사이즈가 큰 객체가(ex. 이미지) 로딩되기 전에도 프록시를 통해 참조할 수 있음(ex. 텍스트(사이즈가 작은 객체인 경우))
  	- 실제 객체의 public, protected 메소드를 숨기고 인터페이스를 통해 노출시킬 수 있음
  	- 로컬에 있지 않고 떨어져 있는 객체를 사용할 수 있음
  	- 원래 객체에 접근에 대해 사전처리를 할 수 있음
  	
  - 단점
  	- 객체 생성 시 한 단계를 거치게 되므로, 빈번한 객체 생성이 필요한 경우 성능 저하가 우려됨
  	- 프록시 내부에서 객체 생성을 위해 thread(쓰레드)가 생성, 동기화가 구현되어야 하는 경우 성능 저하가 우려됨
  	- 로직이 난해해져 가독성이 떨어질 수 있음
  	
  	
  - 종류
  	- 가상 프록시 : 꼭 필요로 하는 시점까지 객체의 생성 연기한 후, '해당 객체가 생성된 것 처럼 동작하도록 만들고 싶을 때' 사용하는 패턴
  	- 원격 프록시 : 원격 객체에 대한 접근을 '제어 로컬 환경'에 존재하며, 원격 객체에 대한 대변자 역할을 하는 객체! 서로 다른 주소 공간에 있는 객체에 대해 마치 '같은 주소 공간에 있는 것처럼 동작하게' 하는 패턴
  	- 보호 프록시 : '주체 클래스에 대한 접근을 제어'하기 위한 경우에 객체 대한 접근 권한을 제어하거나, '객체마다 접근 권한을 달리하고 싶은 경우' 사용하는 패턴
  ```

## (3) 서비스 추상화

- **`추상화 계층을 사용하여 어떤 기술을 내부에 숨기고 개발자에게 편의성을 제공해주는 것이 서비스 추상화(Service Abstraction)`**

**- 일관성 있는(환경의 변화와 관계없이) 서비스 추상화임** 

**- PSA = 잘 만든 인터페이스**

**- Adaptor패턴을 적용해 같은 일을 하는 다수의 기술을 `공통의 인터페이스로 제어`할 수 있게 한 것**

**- HttpServlet을 상속받고 doGet(), doPost()를 구현하는 수고를 들이지 않게 `편의성을 제공`하는 것이 '서비스 추상화의 목표'**

**- *POJO원칙을 철저히 따른 Spring의 기능으로 POJO원칙을 잘 지키게끔 PSA형태의 추상화가 되어있음을 의미**

  ​		***POJO(Plain Old Java Object)** 

  ```
  - 오래된 방식의 간단한 자바 오브젝트
  - 간단하게 getter / setter 를 생각하면 될 것 같음
  ```

- ex) Spring Web MVC(@Controller, @Service .. ), Spring Transaction(@Transactional) , Spring Cache(@Cacheable)

**- 프론트엔드 프레임워크 -> Bootstrap, Foundation, MDL**

# * 프레임워크의 장/단점

**- 장점** 

  **- 효율적 : 시간/비용 절약, 생산성 향상**
  **- 퀄리티 향상 : 다수의 개발자가 사용하며 수정하다 보니 이미 검증된 코드**
  **- 유지보수에 좋음 : 코드가 보다 체계적이라 개발자가 바뀌어도 유지보수에 안정적**

**- 단점** 

  **- 학습시간이 김 : 습득하고 이해하는 데 오랜시간이 걸림**

  **- 의도된 제약 사항 : 자유롭고 유연하게 개발하는 데 한계**
